Practical CBT / ERP / ACT Practice Guide

AI / Tool Loop, Optimization, Productivity, and Compulsive Building

A concrete practice manual for moments when AI, tools, systems, planning, optimizing, researching, and checking become ways to avoid discomfort, uncertainty, or direct output. This guide is broader than thesis work: it applies to academic writing, coding, creative projects, work tasks, personal admin, learning, knowledge management, and productivity systems.

Focus: AI/tool loop first
Method: behavioral experiments
Rule: output before optimization
Tone: firm, non-punitive

Contents

1. Starting principle 2. AI/tool loop 3. Optimization spiral 4. Productivity trap 5. Tool-building loop 6. Checking + researching 7. Always-on availability 8. Combined protocols 9. Real-world examples 10. Logs and templates 11. First week

1. Starting principle

The goal is not to ask yourself better questions or repeat calming phrases. The goal is to run small behavioral experiments that change what your nervous system learns.

Core learning target

I can feel the pull to ask AI, optimize, build, plan, check, or research — and still do the direct output task first.

In practice, this means that the intervention must include an actual behavior: writing a paragraph, sending a draft, making a decision, coding a rough function, completing a boring admin step, cleaning one physical object, or stopping work on time. Self-talk is only useful if it helps you do the behavior while the urge is still present.

The practical CBT shape

1. Notice the trigger.
2. Predict what will go wrong if you do not perform the compulsion.
3. Block or delay the compulsion.
4. Do a small direct-output action.
5. Measure what actually happened.
6. Repeat until the threat prediction weakens.

Direct output vs support behavior

Direct output Support behavior Compulsive version
Writing 250 rough words Checking notes to verify a fact Rechecking notes repeatedly before writing
Coding a minimal working function Reading documentation for one error Reading docs, examples, and comparisons before trying
Sending a simple email Checking the recipient or attachment once Rereading the email five times and asking AI to improve tone
Making one decision Getting one piece of information Prompting AI for alternatives, comparing tools, searching more opinions
Studying one chapter Using a note template Building a better knowledge system instead of studying
Important

Support behaviors are not banned. They become a problem when they are used before direct output as a way to reduce anxiety, increase certainty, or avoid imperfect action.

2. AI/tool loop: the main starting point

The AI/tool loop is especially powerful because it combines reassurance, optimization, productivity, research, and tool-building in one frictionless interface.

What it looks like

  • Asking AI to frame the task before you try.
  • Prompting for better options, then more options, then a comparison.
  • Using AI to judge whether your own idea is acceptable.
  • Moving from one model to another: ChatGPT → Claude → Perplexity → Google → Reddit → back to ChatGPT.
  • Creating prompts, templates, agents, workflows, summaries, and integrations instead of doing the task.
  • Asking AI to make a plan, then another plan, then a more realistic plan, then a tracker.
  • Rewriting the same paragraph, email, outline, or code snippet repeatedly without committing.

The hidden functions

AI behavior Possible function Cost
“Give me 10 ways to frame this.” Avoiding commitment More options, less decision
“Is this good enough?” Reassurance seeking Lower trust in own judgment
“Build me a workflow for this.” Preparatory avoidance System expands, task remains
“Rewrite this again, more precise.” Perfectionistic editing No stopping point
“Compare these three approaches.” Certainty seeking Analysis paralysis

The first rule: output before AI

Rule

AI may respond to your output. AI may not replace the first act of output.

Before-output AI use

“Help me figure out what I should write.”

“Give me the best structure.”

“Generate options before I decide.”

“What’s the optimal way to approach this?”

After-output AI use

“Here is my rough paragraph. Identify the main claim.”

“Here is my draft email. Flag only ambiguity or missing context.”

“Here is my code. Identify the likely bug without rewriting the whole thing.”

“Here is my plan. Find one unrealistic assumption.”

AI exposure practice: 20-minute no-AI commitment

This is the core exercise. Use it with thesis work, job work, personal decisions, coding, writing, or planning.

1. Pick one direct-output task.
   Examples:
   - Write one rough paragraph.
   - Draft one email.
   - Code the first crude version.
   - Choose one option.
   - Fill one form.
   - Make one phone call script.

2. Write the prediction:
   “If I do this without AI first, then ________.”

3. Set a 20-minute timer.

4. During the timer:
   No AI.
   No search.
   No tool changes.
   No comparing.
   No asking for alternatives.

5. Produce the rough version.

6. After the timer, decide:
   Does this actually need AI feedback?
   If yes, ask ONE narrow question about the existing output.

Concrete example: thesis

Compulsive urge: “I need AI to help me frame this theme before I write.”

Prediction: “If I do not ask AI first, the framing will be weak and I will waste the block.”

Practice: Write for 20 minutes beginning with: “My current interpretation is...”

Blocked behaviors: no AI, no theme comparison, no better wording search.

After: If needed, ask: “Here is my paragraph. What is the main claim? Do not rewrite.”

Concrete example: coding

Compulsive urge: “I should ask AI for the cleanest architecture before coding.”

Prediction: “If I start without AI, I’ll build it wrong.”

Practice: Write the dumbest working function or failing test in 20 minutes.

Blocked behaviors: no architecture prompt, no library comparison, no setup refactor.

After: Ask AI only: “Given this function and this error, what is the smallest fix?”

Concrete example: personal admin

Compulsive urge: “I should ask AI to draft the perfect message / compare all options / plan the process.”

Prediction: “If I write it myself, it will sound wrong.”

Practice: Draft the message in plain language in five minutes.

Blocked behaviors: no AI tone rewrite, no searching examples, no overexplaining.

After: Send if clear enough, or ask AI one narrow question: “Is anything unclear or missing?”

AI stopping rules

Rule Practical application
One prompt per problem Ask once. Do not rephrase the same anxiety in new wording.
No cross-model reassurance No asking ChatGPT, then Claude, then another tool to compare.
No “10 alternatives” prompts Ask for one critique of your existing output, not more possibilities.
AI window after output AI opens only after 20–30 minutes of direct output.
Use AI for defects, not identity Ask “what is unclear?” not “is this good / am I doing this right?”

3. Optimization spiral

Optimization becomes compulsive when the standard keeps moving and improvement replaces completion.

The optimization trap

Optimization often starts legitimately. You improve the outline, rename the theme, adjust the folder structure, refine the workflow, choose a better note format, or ask AI for a cleaner version. The trap begins when each improvement creates another possible improvement.

Trap signature

If the task keeps becoming “almost ready,” you are probably optimizing the conditions for output instead of producing output.

Optimization stopping rules

Situation Stopping rule What to do instead
Renaming sections/themes/projects Name is locked for 48 hours Write under the current name
Changing outline/architecture No structure changes during output block Add bracket: [structure idea later]
Improving workflow/system No workflow changes before one output unit Use the current flawed setup
Editing same text repeatedly Two-pass edit maximum Move to next section/task
Tool comparison Use current tool for 7 days Record friction, do not switch

Behavioral experiment: write under the imperfect setup

Prediction:
“If I use the current imperfect setup, the output will be poor / confused / inefficient.”

Experiment:
Use the current setup for one 25-minute output block.
No renaming, restructuring, redesigning, tool changes, or AI alternatives.

Measure:
- Did output exist at the end?
- Was the output unusable or just imperfect?
- Did the missing optimization truly block the task?
- Did the urge decrease after starting?

Practical example: knowledge management

Scenario: You want to improve your note system before working.

Old loop: Open notes → notice inconsistent tags → redesign tags → create summary template → test AI integration → no real task output.

CBT practice: Choose one existing note. Use it as-is. Produce one paragraph, one decision, or one concrete next action from it.

Blocked behavior: no tag cleanup, no template changes, no summary conversion.

After 25 minutes: write down whether the messy note actually prevented output.

4. Productivity trap

Productivity becomes compulsive when planning, tracking, organizing, and system maintenance produce relief instead of output.

Common forms

  • Making a better plan instead of doing the first step.
  • Opening a task manager instead of opening the actual work file.
  • Organizing notes, labels, folders, highlights, tabs, or boards.
  • Creating routines, dashboards, trackers, and review systems.
  • Watching productivity content or reading about methods.
  • Turning CBT itself into a tracker-building project.

The 7-minute planning cap

Rule

Planning is allowed only until it produces the next physical action. Then planning stops.

7-minute planning template:
1. Output I will produce:
2. First physical action:
3. What I am not allowed to do during the block:
4. Timer length:

Then start.

Examples of acceptable outputs

Context Bad target Good target
Thesis Clarify Theme 2 Write 200 words explaining Theme 2’s central claim
Work project Make progress on strategy Draft the first three bullets of the client update
Coding Plan the architecture Write one failing test for the login function
Life admin Organize finances Open the bill and pay it or schedule payment
Learning Build a study system Complete 10 practice problems

Behavioral experiment: unplanned production

Prediction:
“If I do not plan properly, I will waste the session.”

Experiment:
Plan for 7 minutes. Produce for 25 minutes.
No task manager, no organizing, no tracker, no dashboard.

Measure:
- What output exists?
- Was the session actually wasted?
- What planning was genuinely necessary?
- What planning was relief-seeking?

Anti-productivity practice: the plain document rule

When you are tempted to create a system, open the plainest possible tool instead: blank document, notebook page, basic text editor, simple email draft, plain spreadsheet, or paper.

Scenario: You want to make a tracker for your CBT practice.

Old loop: Compare apps → create fields → automate logs → design dashboard → no exposure practice.

CBT practice: Use a single paper line: “Today I delayed AI for 20 minutes and produced X.”

Rule: No system is allowed until seven plain-paper repetitions exist.

5. Compulsive tool-building

Tool-building is one of the most seductive loops because it can be genuinely useful, technically interesting, and easy to defend.

How it fuses with AI + optimization + productivity

The tool-building loop often looks like this:

Discomfort with direct task
↓
“I need a better system/tool/context/workflow.”
↓
Build or improve infrastructure
↓
Short-term relief, stimulation, control
↓
More possibilities appear
↓
The direct task remains undone or expands

The tool quarantine

Rule

Tool ideas are captured, not executed, during output time.

Tool quarantine entry:
Tool idea:
Trigger/discomfort:
What direct task was I avoiding?
Which exact output would this help today?
Can I do 25 minutes without it? Yes/No
Review date:

Three tests before building anything

Test Question If answer is no
Immediate-output test Will this help produce a named output today? Do not build
Current-tool test Have I tried doing it with the current setup for 25 minutes? Do that first
Cost test Will this take less than 20 minutes? Defer to scheduled tool window

Behavioral experiment: current toolchain exposure

Prediction:
“If I do not build/improve this tool, I will be unable to work effectively.”

Experiment:
Use the current imperfect toolchain for 25 minutes.
Produce one direct-output unit.
No integrations, automations, templates, summaries, tagging, prompt libraries, or workflow redesigns.

Measure:
- Was the task impossible?
- What exact friction appeared?
- Could the friction be tolerated for one block?
- Is the tool still needed after the output exists?

Concrete examples

Writing / thesis

Urge: Build a quote extraction workflow.

Practice: Manually paste two quotes into the draft and write interpretation.

Blocked: automation, highlighting redesign, AI extraction pipeline.

Coding

Urge: Create a better dev environment before implementing.

Practice: Implement the smallest ugly version in current environment.

Blocked: config cleanup, tooling comparison, framework switch.

Knowledge management

Urge: Build a better summary template.

Practice: Write a 5-bullet summary in the existing note.

Blocked: template design, tag redesign, AI summary workflow.

Life admin

Urge: Build a finance dashboard before handling bills.

Practice: Pay one bill or list three due dates on paper.

Blocked: spreadsheet redesign, app comparison, categorization.

6. Checking + researching as AI-adjacent loops

Checking and researching become more powerful when AI and search make them instant, endless, and socially frictionless.

Checking through AI

Sometimes AI use is just checking in a new form.

Surface question Actual function Replacement practice
“Is this paragraph okay?” Reassurance Send to one bounded checklist: claim, evidence, transition
“Did I miss anything?” Certainty seeking Name one specific risk; check once later
“Can you verify this again?” Recursive checking Stop after one verification pass
“Compare my answer to best practice.” Perfectionistic benchmarking Use minimum viable standard

Research through AI/search

Research is legitimate only when tied to a specific output gap.

Open-ended research

“I need to understand this better.”

“What are all the approaches?”

“What am I missing?”

“Find the best sources.”

Bounded research

“I need one source for this claim.”

“I need the definition of this term.”

“I need to check whether this API supports X.”

“I need the deadline for this form.”

Research stopping rule

Before research:
Write the output gap in one sentence.

During research:
Maximum 25 minutes.
Maximum 3 sources or 3 pages.

After research:
Return immediately to the output.
No new branches unless they block the current output.

Behavioral experiment: one-source repair

Prediction:
“If I do not research broadly, I will miss something important.”

Experiment:
Use only one targeted source to repair one existing gap.
Then return to output.

Measure:
- Was broad research actually needed?
- Did one targeted repair move the task forward?
- Did broad curiosity try to take over?

7. Compulsive availability

The AI/tool loop is not only about what you do during work. It also keeps you mentally attached after work should be over.

What always-on looks like

  • Reopening AI or notes at night because an idea appeared.
  • Checking messages, comments, dashboards, or task boards “just in case.”
  • Mentally redesigning systems while walking, showering, eating, or lying in bed.
  • Keeping the work problem alive because stopping feels unsafe.
  • Using AI after hours because it feels like a low-friction way to keep working.

Shutdown protocol

1. Write what exists now.
2. Write the next exact action.
3. Write what remains unresolved.
4. Park one sentence for tomorrow.
5. Close the work surface.
6. No AI, search, notes, dashboards, or work tools after shutdown.

Example shutdown note

What exists: Rough paragraph on Theme 2 / basic function implemented / email draft started.

Where I stopped: Need transition after paragraph 2 / bug appears in validation / need one date for the email.

Next exact action: Open the file and write the transition beginning “This matters because...” / run test X / check the date.

Allowed unresolved: final wording, optimal structure, better tool, source completeness, emotional certainty.

Evening urge practice

Trigger:
“I should just ask AI / check notes / fix the system now.”

Practice:
Write one paper line:
“Tomorrow: ________.”

Then:
No screen.
No AI.
No tool.
Return to non-work activity for 10 minutes.

Measure tomorrow:
Was the idea still important?
Could I resume from the note?

8. Combined protocols for mixed loops

In reality, the compulsions combine. You do not experience “pure AI dependence” or “pure productivity.” You experience chains. These protocols target the chains.

Chain A: AI → optimization → productivity

Pattern: Ask AI for framing → get several options → redesign plan → create tracker → no output.

Intervention: Choose one option arbitrarily enough, write for 20 minutes, then evaluate from output.

Protocol A:
1. Ask no AI before output.
2. Pick current best version.
3. Lock it for 20 minutes.
4. Produce rough output.
5. Only after output: one critique question allowed.
6. No tracker/system creation.

Chain B: tool-building → research → AI

Pattern: Build better note system → realize you need better sources → ask AI to summarize them → start building summary workflow.

Intervention: Use current note, write the target output, park all tool/research ideas.

Protocol B:
1. Write the tool idea in quarantine.
2. Open current imperfect material.
3. Produce one output unit for 25 minutes.
4. Mark gaps in brackets.
5. After output, choose at most one support action.

Chain C: checking → AI reassurance → editing loop

Pattern: Recheck draft → ask AI if it works → rewrite → ask again → compare versions.

Intervention: One checklist pass, one edit pass, stop.

Protocol C:
1. Define checklist before reading:
   - Is the claim clear?
   - Is there evidence?
   - Is the next action obvious?

2. Read once.
3. Make one edit pass.
4. Do not ask AI for reassurance.
5. Move to next output or stop.

Chain D: always-on → late-night AI/tooling

Pattern: Idea appears at night → open AI → generate plan → tweak workflow → sleep delayed.

Intervention: one-line capture on paper, no screen, tomorrow-only rule.

Protocol D:
1. Write one paper line.
2. Add “review tomorrow at ___.”
3. No AI/tool/search after shutdown.
4. If still relevant tomorrow, handle in support block.
5. Log whether late-night action was actually necessary.

9. Real-world examples

These examples show how to apply the same principles outside thesis work.

Example 1: writing an important email

Old loop: Draft email → ask AI to improve tone → compare versions → worry it sounds too formal → ask again → reread → delay sending.

New practice:

1. Draft email in plain language in 5 minutes.
2. Check only: recipient, main ask, attachment.
3. Optional AI question: “Is anything unclear or missing?”
4. Make one edit pass.
5. Send.
6. Do not reread sent email.

Exposure: send a clear but imperfect email without optimizing tone.

Example 2: starting a coding task

Old loop: Ask AI for architecture → compare libraries → refactor setup → watch examples → no working code.

New practice:

1. Write one failing test or crude function.
2. Use current tools only.
3. No architecture discussion for 25 minutes.
4. If stuck after trying, ask AI for the smallest next fix.
5. Stop after the fix or after one test passes.

Exposure: tolerate ugly first implementation.

Example 3: organizing personal finances

Old loop: Build spreadsheet → compare budgeting apps → categorize transactions → feel productive → avoid paying one bill or making one call.

New practice:

1. Identify one concrete financial action.
2. Do it before any spreadsheet or app work.
3. Example: pay bill, call provider, list due dates.
4. After action, decide if system work is still necessary.
5. If yes, cap system work at 20 minutes.

Exposure: do a real financial action while the system is imperfect.

Example 4: learning a new topic

Old loop: Ask AI for learning roadmap → compare courses → build notes template → collect resources → do no practice.

New practice:

1. Do 20 minutes of direct practice.
2. Solve problems, write explanation, attempt exercise.
3. No roadmap, no resource comparison, no note system.
4. After practice, ask one specific question about what you got wrong.

Exposure: be a beginner without preparing perfectly.

Example 5: creative project

Old loop: Ask AI for ideas → generate variations → collect references → create moodboard → adjust tools → no draft.

New practice:

1. Make one ugly first version in 20 minutes.
2. No reference search.
3. No tool changes.
4. No alternate concepts.
5. After the ugly version exists, choose one improvement.

Exposure: create before curating possibilities.

11. First week: simple practice plan

Do not attempt to fix every behavior at once. The first week is about building evidence that output can happen before AI, optimization, productivity systems, or tools.

Day Practice Blocked behavior Success criterion
Day 1 20 minutes output before AI AI prompting, option generation Rough output exists
Day 2 25 minutes under current structure Optimizing, renaming, restructuring Output under imperfect structure
Day 3 7 minutes planning + 25 minutes output Productivity systems, task organizing Planning stayed capped
Day 4 Use current toolchain for 25 minutes Tool-building, integrations, templates Output without tool change
Day 5 Draft first, research one gap later Open-ended research One paragraph before source search
Day 6 One-pass edit/check Recursive checking, AI reassurance One check/edit pass only
Day 7 Shutdown exposure Always-on availability Stopped with restart note

Minimum version for bad days

Minimum viable practice

Produce one direct-output sentence before AI, search, tool-building, optimizing, or organizing. Then log the urge. That counts.

What not to do

  • Do not build a CBT dashboard.
  • Do not create a sophisticated AI prompt protocol.
  • Do not compare productivity methods.
  • Do not make a complex exposure hierarchy before doing one exposure.
  • Do not use this guide as reading material instead of practice.
Final warning

The loop will try to absorb the treatment. It will say: “Let me optimize this plan, make a tracker, ask AI to personalize it, create a better system, and then start.” That is the same behavior. Start with one ugly output block.
