Practical CBT Guide · AI / Tool Loop Focus

AI, Tool-Building, Optimization & Productivity Loops

A concrete practice manual for interrupting AI/tool compulsions that blend with optimization, research, productivity systems, and “just one more improvement” behavior. The goal is not to ban tools. The goal is to stop using tools to avoid uncertainty, commitment, emotional discomfort, or imperfect output.

Use this as a working practice sheet. It is not a diagnosis or a substitute for therapy.

The practical target

Produce the real-world output before improving the system around the output.

The loop is not “I use AI” or “I build tools.” The loop is:

discomfort / uncertainty / pressure
→ open AI, build tool, compare options, optimize workflow, research more
→ short-term relief and sense of progress
→ less tolerance for deciding or producing without support
→ stronger urge next time

CBT practice means creating repeated situations where you feel the urge, do not perform the usual safety behavior, and take a concrete action anyway.

1. Recognize the AI/tool loop in real life

Common forms

LoopWhat it sounds likeWhat it often avoids
AI reassurance loop“Let me ask one more prompt to make sure this is right.”Trusting your own provisional judgment.
Multi-model comparison“I’ll ask ChatGPT, then Claude, then compare.”Committing to one version.
Prompt refinement loop“The answer would be better if I phrased the prompt better.”Using an adequate answer and moving on.
Tool-building loop“If I build this workflow, future work will be easier.”Starting with imperfect tools now.
Research loop“I need one more source / tutorial / benchmark / example.”Making an imperfect decision with current knowledge.
Productivity cockpit loop“I need a better dashboard, tracker, template, tags, database.”Doing the task without feeling fully organized.
Optimization loop“This could be cleaner, faster, more elegant, more reusable.”Accepting local imperfection.

The functional test

Before using AI, building a tool, researching, or optimizing, answer this in writing:

What exact deliverable will this produce or change within the next 20 minutes?

If the answer is vague, the behavior is probably serving discomfort regulation.

Vague answer

“It will help me think better.”

“It will make the system cleaner.”

“It will help future me.”

“It will clarify the whole thing.”

Concrete answer

“It will identify one bug in this function.”

“It will improve the transition in paragraph 3.”

“It will extract three quotes for this section.”

“It will decide between option A and B by 14:30.”

2. The core protocol: Output → Support → Stop

This is the basic rule. Use it across work, study, writing, coding, planning, admin, and personal decisions.

1. Output first:
   Produce a rough version using current tools and current knowledge.

2. Support second:
   Use AI / research / checking / optimization only to improve the existing output.

3. Stop third:
   End after one bounded support pass. Do not restart the loop.

Why this works

It does not rely on motivation. It changes the order of behavior. The compulsion wants support first; the practice forces output first.

3. AI use rules that are concrete enough to follow

Rule A: Draft-first AI

You may use AI only after you have created your own version.

Before AI:
- Write the paragraph.
- Choose the option.
- Draft the email.
- Sketch the plan.
- Make the decision.
- Write the code attempt.
- State the current interpretation.

Then AI may comment on what exists.

Practical examples

SituationCompulsive useFunctional use
Writing“Give me 10 possible framings.”“Here is my 300-word draft. Identify the main claim and one unclear sentence. Do not rewrite.”
Coding“Build the whole architecture for me.”“Here is my failing function and error message. Identify the likely bug. Give one fix only.”
Decision“Which option is best? Ask again with more nuance.”“I chose option A. List one risk I should mitigate before proceeding.”
Planning“Create the perfect system for managing this project.”“Here are my next three tasks. Put them in a reasonable order. Do not add new tasks.”

Rule B: One prompt → one action

Every AI interaction must end in a real-world action, not another prompt.

Prompt:
Ask one bounded question.

Answer:
Extract one decision or correction.

Action:
Implement it immediately.

Stop:
Close AI before asking a second version.
Example: writing loop

Old loop: Ask for better wording → ask for more academic version → ask for clearer version → compare → lose trust → ask another model.

New practice: Ask: “Which sentence is least clear?” → revise that sentence → close AI.

Example: coding loop

Old loop: Ask for architecture → ask for refactor → ask for best practices → redesign the project.

New practice: Ask: “What is the smallest failing assumption in this function?” → test that one thing → close AI.

Rule C: No model-shopping

Do not ask multiple AI systems the same question to feel more certain.

Allowed:
Ask one model one bounded question.

Not allowed:
Ask ChatGPT, then Claude, then Gemini, then compare answers,
unless the task explicitly requires independent expert comparison.

Exception standard

Model comparison is allowed only if all three conditions are true:

  • The decision is genuinely high-stakes.
  • You write the comparison criteria before asking.
  • You stop after the planned comparison and make a decision.

4. Tool-building rules

The build/no-build decision tree

1. Is there a deliverable due or active today?
   If yes → produce it first.
   If no → continue.

2. Did the missing tool block work today, or merely seem useful?
   If merely useful → do not build.

3. Can the tool be built in 20 minutes?
   If no → write it on the Tool Parking Lot.

4. Will it serve at least two specific future deliverables?
   If no → do not build.

5. Can you define “done” before starting?
   If no → do not build.

6. Build only in a scheduled tool slot.
   Never build during output time.

Red flag

“This will make everything easier later” is not a sufficient reason. It is usually the tool-building compulsion speaking.

Tool Parking Lot template

When the urge to build appears, capture it without acting on it.

Tool idea:
Trigger:
What discomfort did this promise to reduce?
Which exact deliverable would it help today?
Can I complete it in 20 minutes?
What happens if I do not build it for 48 hours?
Review date:

Example

Tool idea:
Automated highlight-to-summary extractor.

Trigger:
Felt overwhelmed by scattered notes.

Discomfort:
Fear that I cannot synthesize without a cleaner system.

Exact deliverable today:
None. It might help generally.

20-minute build?
No.

48-hour delay prediction:
I’ll be less efficient and lose insight.

Action:
Do not build. Write 200 words with current notes. Review Friday.

20-minute ugly tool rule

If a tool is truly necessary, build the ugliest version that solves today’s bottleneck.

Allowed:
A plain text checklist.
A simple table.
A manual copy/paste template.
A one-off script that solves one task.

Not allowed:
Architecture.
Reusable framework.
Dashboard.
Integration.
Automated pipeline.
Perfect taxonomy.
Styling, tags, metadata, sync, multi-device support.
Example

Old loop: Build a full note extraction pipeline with summaries, tags, embeddings, and Claude context.

New practice: Make a 3-column table: Source / Useful quote / Why it matters. Fill five rows. Stop.

5. Optimization rules

The local-improvement rule

Optimization is allowed only locally, not systemically, during work time.

Local optimization:
Improve this paragraph.
Fix this bug.
Clarify this decision.
Rename this one heading after draft exists.

Systemic optimization:
Rebuild workflow.
Redesign folder structure.
Change the whole outline.
Create new tags.
Rewrite the whole prompt system.
Switch apps.
Create a new dashboard.

Rule: systemic optimization is never allowed before output.

Optimization exposure drill

1. Pick a task.
2. Intentionally use the current imperfect system.
3. Produce output for 25 minutes.
4. Write every optimization urge on paper.
5. Do not act on any optimization urge.
6. After 25 minutes, review:
   - Which urges still matter?
   - Which were anxiety?
   - Did the imperfect system actually prevent output?

Examples

  • Use the current messy notes to write one paragraph.
  • Use the current folder structure to find one document, not reorganize the folder.
  • Use the existing prompt once, not design a better prompt library.
  • Use your current task list, not migrate to a new app.

6. Research-loop rules

Research must answer a question created by output

Research before output often expands uncertainty. Research after output can repair a real gap.

Bad:
Research to feel ready.

Good:
Write first, find the exact gap, then research that gap.

Question standard

Before researching, write a question narrow enough to answer in 25 minutes.

Too broad

“How do people think about productivity and AI?”

“What is the best note-taking system?”

“What does the literature say?”

Narrow enough

“Find one source supporting X.”

“Check whether this claim is contradicted by Y.”

“Find one example of Z.”

25-minute research cap

Timer: 25 minutes.
Source cap: 3 sources.
Output required: one written note or one change to the deliverable.
Stop when timer ends, even if unresolved.
Example

Old loop: Read articles, follow citations, open tabs, collect more material, delay claim.

New practice: Write the claim first. Search for one source. Add one sentence or bracket. Stop.

7. Productivity-loop rules

Planning is limited to producing the next action

Planning question:
What will I produce in the next 25 minutes?

Not allowed:
What is the full system?
What is the perfect workflow?
How should I organize the whole project?
How do I prevent all future friction?

7-minute planning cap

Minute 0–2:
Choose one deliverable.

Minute 2–5:
List the first three physical actions.

Minute 5–7:
Open the work surface.

Minute 7:
Start producing, even if plan is incomplete.

Productivity substitution detector

If you switch from the hard task to any of these, pause:

  • task list editing
  • calendar rearranging
  • file cleanup
  • app setup
  • template creation
  • dashboard checking
  • progress tracking
  • AI planning
  • workflow research
Question:
Am I producing the deliverable, or producing the feeling of being organized?

8. Urge handling: what to physically do when the pull hits

The 90-second interruption

Use this before opening AI, starting a tool build, researching, or optimizing.

0–15 seconds:
Hands off keyboard or phone. Feet on floor.

15–30 seconds:
Name the behavior:
“This is the urge to ask/build/optimize/research.”

30–45 seconds:
Locate body sensation:
chest / throat / stomach / face / hands / head.

45–60 seconds:
Rate intensity:
0–10.

60–90 seconds:
Write one sentence:
“If I do not do this, I predict ________.”

Then set a 10-minute delay.

This is not a mantra. It is a behavioral interruption: stop motor action, identify the urge, write a prediction, delay the compulsion.

10-minute delay with action

For 10 minutes, do not:
- ask AI
- build the tool
- optimize
- research
- check
- reorganize

During those 10 minutes, do:
- write one sentence
- make one decision
- edit one paragraph
- test one function
- send one draft
- fill one row of a table
- create one imperfect output object

Important

The delay must contain action. If you only sit and suffer, you may learn endurance but not workability. The target learning is: “I can act while the urge is present.”

9. Exposure hierarchy for AI/tool loops

Start low. Repeat until the urge drops or becomes easier to carry. Then move up.

2Write a rough paragraph before asking AI.AI-before-draft.Paragraph exists first.
3Ask AI one question only, then implement one answer.Re-prompting.AI closed after one action.
4Do one task with current tools.Tool improvement.Output created without changing system.
5Use one source only to answer a narrow gap.Research expansion.Gap addressed or bracketed.
6Make a decision without AI comparison.Model-shopping.Decision recorded and acted on.

10. Concrete scenarios and exact responses

Scenario 1: “I need to build a better summary system.”

Old path:
Open notes → feel overwhelmed → design summary format → ask AI for best format →
test templates → adjust categories → no actual output.

CBT path:
1. Write prediction:
   “If I do not build this system, I will not be able to synthesize.”

2. Delay tool-building 25 minutes.

3. Use current notes to produce one crude output:
   - five bullet summary
   - one paragraph
   - one decision
   - one table row

4. After 25 minutes, check:
   Was synthesis impossible, or uncomfortable?

5. Park the tool idea for weekly review.

Scenario 2: “I should ask AI to clarify this before I start.”

Old path:
Ask AI → get options → ask for refinement → compare →
become less certain → ask another model.

CBT path:
1. Write your own 5-sentence version first.

2. Then either:
   A. Continue without AI, or
   B. Ask one narrow question:
      “What is the least clear sentence? Do not rewrite.”

3. Fix one sentence.

4. Close AI.

Scenario 3: “This workflow is inefficient.”

Old path:
Notice friction → redesign workflow → research tools →
migrate system → lose time → feel briefly in control.

CBT path:
1. Name the friction:
   “Finding quotes is slow.”

2. Ask:
   “Can I still complete today’s task manually?”

3. If yes:
   complete it manually.

4. If no:
   build the ugliest 20-minute workaround only.

5. No migration, no dashboard, no integration.

Scenario 4: “I need more research before deciding.”

Old path:
Open browser → follow links → collect sources → delay decision.

CBT path:
1. State the decision:
   “I need to decide between A and B.”

2. Write what you already know.

3. Set one narrow question:
   “What is the strongest reason against A?”

4. Research 25 minutes, 3 sources max.

5. Decide or set a timed next step.
   Do not keep browsing.

Scenario 5: “I want to reorganize everything because it feels messy.”

Old path:
Clean notes → change categories → create tags → restructure folders →
feel temporary relief → avoid the hard output.

CBT path:
1. Write:
   “Messy does not mean blocked.”

2. Choose one messy item and use it.

3. Produce one output from it.

4. Put reorganization urge on paper:
   “Maybe reorganize X later.”

5. Leave the system messy for 48 hours.

12. Weekly review that does not become another system

Do this once per week. Maximum 20 minutes. Paper or plain text only.

1. Which loop cost me the most time this week?

2. Which safety behavior did I delay successfully?

3. Which prediction was disproven?

4. Which support behavior was actually useful after output?

5. Which support behavior was mostly anxiety relief?

6. What is the one exposure I will repeat next week?

Do not do this

Do not build a dashboard, scoring system, database, tracker, automation, or AI assistant for this practice. That would turn the treatment into the compulsion.
